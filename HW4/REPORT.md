# REPORT

## Суть приложения

DROP LOTTERY — это децентрализованная лотерея на Ethereum. Пользователь подключает кошелек, отправляет фиксированную сумму для участия, после чего контракт собирает пул участников и по истечении интервала выбирает победителя. Случайность берется из Chainlink VRF, а интерфейс показывает состояние лотереи, таймер, призовой фонд и журнал событий.

## Технологии

### Solidity

Solidity использовался для написания смарт-контракта `Raffle.sol`, который хранит участников, баланс, состояние раунда и логику выбора победителя. Для Ethereum это стандартный язык.

### Hardhat

Hardhat использовался как среда разработки контрактов: компиляция, локальная нода, тесты и скрипты деплоя. По сравнению с Foundry он проще для проекта, который уже строится вокруг TypeScript и npm-скриптов, а по сравнению с Truffle он современнее и удобнее в настройке локальных сценариев.

### Chainlink VRF v2.5

Chainlink VRF нужен для честной случайности при выборе победителя. Он был выбран вместо псевдослучайности на основе `block.timestamp` или `blockhash`, потому что такие подходы уязвимы для манипуляций, тогда как VRF дает проверяемый и внешне подтвержденный источник случайного числа.

### Chainlink Automation v2

Chainlink Automation отвечает за автоматический запуск розыгрыша, когда прошло нужное время и есть участники. Это удобнее ручного вызова админом и безопаснее кастомного централизованного бэкенда с cron-задачей, потому что логика инициирования раунда остается в экосистеме Web3.

### Ethers.js v6

Ethers.js использовалась и на фронтенде, и в Hardhat-скриптах для чтения данных контракта, отправки транзакций и подписки на события. Она легче и компактнее, чем `web3.js`, а версия v6 дает более строгую типизацию и современный API.

### React 18

React используется для интерфейса приложения: страницы, виджеты, кнопки и связка UI с блокчейн-данными. По сравнению с Vue он чаще встречается в Web3-проектах и имеет больше готовых интеграций вокруг кошельков, а по сравнению с чистым DOM-кодом позволяет проще собирать интерфейс из компонентов.

### Vite

Vite использовался как инструмент сборки и локальной разработки фронтенда. Он запускается быстрее, чем Webpack, и требует меньше конфигурации для React-проекта, поэтому хорошо подходит для небольшого приложения.

### Tailwind CSS

Tailwind CSS использовался для стилизации интерфейса и сборки неоновой киберпанк-визуальной темы без отдельного большого слоя CSS-компонентов. По сравнению с Bootstrap он дает меньше навязанных готовых шаблонов, а по сравнению с CSS Modules позволяет быстрее собирать интерфейс прямо в JSX.

### Zustand

Zustand использовался для хранения состояния лотереи и журнала событий на клиенте. Он заметно проще Redux по объему кода и порогу входа, а для небольшого приложения без сложной асинхронной бизнес-логики этого подхода достаточно.

### React Router v6

React Router отвечает за маршрутизацию между главной страницей и страницей `404`. Для SPA это стандартное и простое решение; по сравнению с самописной маршрутизацией он лучше масштабируется, а по сравнению с Next.js не требует полной смены архитектуры проекта.

### React Hot Toast

React Hot Toast использовался для быстрых уведомлений о статусе транзакций: отправка, ожидание подтверждения, успех и ошибка. По сравнению с самописными alert-компонентами он быстрее внедряется и по сравнению с более тяжелыми UI-библиотеками не тянет за собой лишнюю инфраструктуру.

## Заключение

Работа над проектом показала, что даже небольшой dApp быстро превращается в задачу на стыке нескольких областей: контракты, локальная блокчейн-среда, кошелек, обработка событий и фронтенд-состояние. Лучше всего в проекте себя показали Hardhat и Ethers.js за предсказуемый DX, а Chainlink оказался полезным именно там, где нужна честная автоматизация без изобретения собственных небезопасных механизмов. В целом стек оказался удачным для проекта: он достаточно современный, чтобы показать реальные подходы из Web3 разработки, и при этом остается удобным для отладки.
